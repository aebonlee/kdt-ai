## 1. ⚠️ 분류 정정 — otherdeep가 아니라 담당 과목 체계로 반영했다

**지시는 "담당 외 계열 → `otherdeep.js` / `otherexams.js`"였으나, 그대로 하면 데이터가 깨진다.**

확인한 사실:
- `prompt`는 **curriculum.js의 담당 18과목 중 하나**다 (code `8-1`, `transformer` 8-2 바로 앞, 1일 과정)
- 이미 담당 체계 전반에 콘텐츠가 존재 — `lectureexamples3['prompt-1']` 17카드, `exams.prompt`(담당 12과목 평가기준), `examsAlt.prompt`(박병선 강사안), 퀴즈, `extrastudy.js`
- `otherdeep`·`otherexams`·`othercontent` 어디에도 `prompt` 키가 **없다**

지시의 근거였던 "대표 담당이 아니다"는 **이번 4기에 대표 배정 세션이 없다**는 뜻으로는 맞다(`sessions`에 prompt 0건 확인). 그러나 이 사이트의 담당/담당외 구분은 **누가 그날 강의하느냐가 아니라 curriculum 18과목 소속 여부**다. otherdeep에 넣었다면:
- 같은 과목이 담당 18과목과 담당외 12과목에 **동시 등장**해 중복
- 스모크 `실습교안 페이지 31 = 개요+18과목+기타개요+담당외 11과목`과 `평가기준 18 = 담당 12 + 참고 6`이 **양쪽 다 깨짐**

→ **transformer와 동일하게 `lectureexamples3['prompt-1']` + `exams.prompt`로 반영**했다. 축적 자산으로 남기라는 지시의 취지(내년 기수 대비)는 담당 과목 체계에 두는 편이 오히려 더 잘 달성된다 — 강의안·퀴즈·평가기준·추가학습이 모두 연결되기 때문이다.

## 2. ⚠️ 팀 인원 — 이번에는 불일치가 아니라 "강사별로 실제로 다름"

transformer에서는 채점표 제목만 2인인 **오타**였지만, 이 과목은 **실제로 운영 방식이 갈린다.**

| 출처 | 팀 인원 |
|---|---|
| **임성열 정본** — 표지 부제 "3인 팀 실습 사전 학습 자료" | **3인 1팀** |
| **임성열 정본** — SESSION FLOW "팀 구성 : 3인 1팀" | **3인 1팀** |
| 임성열 교수자용 채점표 | 팀 인원 **언급 없음** (불일치 없음) |
| **하만석 강사안** — "2인 1조로 시나리오를 기획", "제출 단위: 조별 1부(2인 1조)", "팀원 2인 역할·기여" | **2인 1조** |
| **박영웅 강사안** — "개인 실습 / AI 도구 자유 선택" | **개인** |
| 윤재성 강사안 — "실습 형태: 조별 실습" | 인원 미명시 |
| 기존 `exams.js` | 2인 (구 버전) |

→ `exams.prompt`는 **정본 기준 3인 1팀**으로 갱신하되, notes에 **"실습교수별로 다름(하만석 2인 1조, 박영웅 개인) — 반별 안내 우선"**을 명시했다.
**7/22 담당이 박창렬·김범준·김영희·하만석이므로 실제 반에서는 2인 1조로 운영될 가능성이 높다** — 대표 확인 필요.

## 3. 강사별 버전 차이 (정본 우선, 차이는 병기)

| 강사 | 과제 구성 | 배점 |
|---|---|---|
| **임성열 (정본)** | B2B 서비스 기획 → 기법 매핑 → 프롬프트 스크립트, 템플릿 3종 | 학생 배포용 rubric에 **배점 수치 없음** — Biz가치/기술이해도/수업충실도 3항목의 확인 포인트만 제시 |
| 박병선 (examsAlt 기반영) | 조별 프롬프트 개선 전·후 비교 | 25/25/30/20 = 100점 |
| 하만석 | 문제 1~6 노트북, Prompt/Context/Harness 코드 확인 | ipynb 1개 제출, 2인 1조 |
| 박영웅 | v0→v5 요소 누적 개선, 마크다운 기록 제출 | 개인, 채점 포인트 기술 |
| 윤재성 | 이커머스 시장진출 보고서, 초기→개선 3단계 | 조별, 당일 마감 |

**정본에 배점 수치가 없다는 점이 중요하다.** 기존 `exams.prompt`의 배점(30/30/40 + 40/30/30)은 SKALA 공통 평가양식에서 온 것으로 판단해 **배점은 그대로 두고**, 각 항목의 `desc`만 정본의 확인 포인트 문구로 교체했다. 배점을 정본에 맞춰 지운다면 오히려 정보가 줄어든다.

`examsAlt.prompt`(박병선 강사안)는 **변경 없음** — 타 강사판 3건 유지(스모크 통과).

## 4. 반영 카드 (prompt-1 +6, 예제 208→214)

기존 17카드 진단 결과 **CoT·ReAct·Chaining·RICE·LLM-as-Judge 등은 이미 충실**했고, 정본의 5기법 중 **SC·Harness·Memory가 빈칸**이었다.

| 카드 | 빈칸 근거 |
|---|---|
| Self-Consistency 제대로 쓰기 — 가짜 SC와 진짜 SC의 갈림길 | 기존에 SC 단독 카드 **없음**. "다시 확인해줘"가 SC가 아닌 이유를 관점 분리로 대비 |
| Memory & Compaction — 무엇을 기억하고 무엇을 요약할지 미리 정하기 | 기존 "Context 4부품"에 히스토리는 있으나 **보존 목록 설계는 없음** |
| Harness Engineering — 거대한 한 덩어리 프롬프트를 단계로 쪼개기 | 기존 "AI 하네스 폴더"는 파일·규칙 관리이고, 정본의 **추출→분석→검증 파이프라인 분리는 없음** |
| 기법 선택 근거 세우기 — 서비스 특성에 따라 무엇을 왜 고르는가 | 서비스 특성 → 기법 매핑 및 근거 문장 틀. 기존 없음 |
| B2B AI 서비스 기획 실습 — 3개 템플릿과 4단계 기획 프레임 | 정본 실습 구조 자체. 기존 없음 |
| 발표 전 자가 점검 — 채점자가 무엇을 보는가 | 기법별 점검표 + 3항목 확인 포인트. 기존 없음 |

## 5. 정답 노출 없이 처리한 방식

채점표 첫 줄: "이 문서는 훈련생에게 배포하지 않고 채점용으로만 사용합니다." → **직전 transformer 작업과 동일한 선.**

이 채점표는 transformer 것과 달리 **실제 정답이 대량으로 들어 있어** 훨씬 조심스럽게 다뤘다.

| 넣은 것 (학생 배포용 정본에 이미 공개) | 뺀 것 (채점키 전용 = 정답) |
|---|---|
| 5개 기법의 패턴 예시(✕/✓ 대비) — 정본 본문에 그대로 있음 | 템플릿 1의 **정답 수치**(크레딧 금액·구독료·가동률·티켓번호) |
| 템플릿 3종의 이해관계자·가치제안·핵심기법·제공자료 개요 | 템플릿 1의 **담당자 이름**(현·전임) 및 변경 이력 |
| "리스크 조항 5개 삽입", "이상거래 5유형 삽입" 같은 **개수**와 정본이 이미 명시한 유형명 | 템플릿 2의 **조항 번호와 리스크 상세 근거** |
| 기법별 확인 포인트("실습 시 이 기법을 확인하는 방법") | 템플릿 3의 **거래ID 5건과 각 이상 유형 매핑** |
| GRADING RUBRIC 3항목의 확인 포인트 | 채점 기준 예시(상/중/하 밴드), 교수 재량 운영 권고 |

- **자동 검증 실시**: 채점키 고유 문자열 21종(금액·이름·티켓번호·거래ID 접두사·조항번호·거래처코드 등)을 카드 JSON에서 grep → **누출 0건 확인**
- 템플릿 3 카드에는 정답 대신 **"금액 순 정렬만으로는 놓치는 유형이 있으니 시각·거래처 등록일·평소 금액대·금액 자릿수 등 여러 축으로 훑어라"**는 **접근법 힌트**만 넣었다. 채점표가 지적한 학생들의 전형적 실패 패턴을 학습 포인트로만 전환한 것이다.
